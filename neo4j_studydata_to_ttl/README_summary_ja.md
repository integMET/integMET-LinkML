# Neo4j JSON to RDF Converter and TTL to RDF-config YAML Generator

---
# Neo4j Study データの RDF 変換パイプライン

本処理系は、Neo4j に蓄積された IntegMet の研究データを、意味的に整理された再利用可能な RDF 資産へ変換するための一連のパイプラインである。

---

## 概要

変換の対象は、Neo4j 上で管理されている Study（研究データセット）とその関連情報である。具体的には、生物種情報、主題語注釈、GO 注釈、分析グループ、分析タイプなどを含む部分グラフが対象となる。このパイプラインは、**JSON 抽出 → RDF 化 → Ontology 生成 → 設定 YAML 生成**という四段階で構成される。

---

## ステップ 1：JSON 抽出（Neo4j → 中間表現）

まず、Neo4j から Study を中心とした部分グラフを JSON として取り出す。この JSON は後続の RDF 化処理における中間表現として機能し、グラフ構造を保ちながらデータを一時的に直列化する役割を担う。

---

## ステップ 2：RDF 化（中間表現 → RDF リソース）

次に、JSON 中間表現をもとに、各 Study を RDF リソースとして構築する。この段階では、以下の情報が体系的に付与される。

- **研究識別子・由来情報**：各 Study を一意に識別するための URI と来歴情報
- **LLM による付加情報**：要約、所見、分類などの自動生成メタデータ
- **外部語彙への参照**：MeSH、GO、NCBI Gene、OMIM などの外部知識資源へのリンク
- **生物種注釈**：対象生物種を注釈セットとして整理した情報
- **分析グループ・測定手法情報**：分析グループ単位で測定技術と結び付けた情報

出力形式は **Turtle、N-Triples、JSON-LD** の三形式が生成される。

本ステップの特徴は、単なるデータ形式の変換にとどまらず、Study を「何の研究で、どのような注釈が付き、どのような分析が行われたか」という観点から機械可読な形で再定義している点にある。すなわち、Neo4j 内部で管理されていた研究情報を、外部公開・他システム連携・知識統合に耐える意味論的データへと昇格させる標準化工程として位置付けられる。

---

## ステップ 3：Ontology 生成

RDF インスタンスデータの生成と並行して、それを解釈するための Ontology が Turtle 形式で出力される。Ontology は、RDF データ中で使用されるクラスおよびプロパティの定義を提供し、インスタンスデータとの対応関係を規定する。

---

## ステップ 4：RDF-config 用 YAML 生成（Ontology + インスタンス → 設定記述）

最終段階では、生成済みの Ontology とインスタンスデータを再度読み込み、後続のデータ利用・設定作業に必要な RDF-config 用 YAML を自動生成する。具体的には、Ontology 上で定義されたクラスと、実データ上で実際に使用されている型・述語の両方を参照しながら、**prefix 定義、主語モデル、述語とオブジェクトの対応、説明文**を含む `model.yaml` および `description.yaml` の初稿を組み立てる。

これにより、RDF 本体に加えて、後続処理への接続に適したモデル記述も得られる。

---

## まとめ

本パイプライン全体は、Neo4j の Study 中心データを段階的に整備し、研究データを意味論的に整理された再利用可能資源へ変換するものである。各ステップが前段の出力を入力として受け取る構成となっており、最終的には RDF データ・Ontology・設定 YAML という三種の成果物が一貫した処理の中で得られる。
