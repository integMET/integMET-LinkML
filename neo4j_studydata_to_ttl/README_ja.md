# neo4j_studydata_to_ttl 

リポジトリ： integMET/integMET-LinkML — `neo4j_studydata_to_ttl` ディレクトリ

---

## 1. 概要

本ディレクトリに収録されているプログラム群は、グラフデータベース Neo4j に格納された学術研究（Study）データを、セマンティックウェブ標準規格である RDF（Resource Description Framework）形式へ変換し、さらにその変換後データから RDF-config（YAML形式の設定ファイル）の初期草案を自動生成することを主目的としたツール群である。

## 3. 主要スクリプト1：`neo4j_study_json_to_rdf_with_ontology.py`

### 3-1. 処理の概要

本スクリプトは、integMET Neo4j データベースから Cypher クエリによって抽出した JSON 形式のグラフデータを入力として受け取り、これを RDF リソースとして表現するとともに、対応するオントロジーファイルを自動生成する。利用ライブラリは Python の `rdflib` のみであり、依存関係は最小限に抑えられている。

### 3-2. 入力データの構造

入力 JSON ファイル（`neo4j_query_table_data_2026-4-3.json`）は、`neo4j_study_json_to_rdf.cypher` に定義された Cypher クエリを Neo4j データベースに対して実行することによって取得したものである。同ファイルは Neo4j の標準グラフ出力形式に準じており、最上位に `graph_json` オブジェクトを持ち、その下に `nodes`（ノード）と `relationships`（エッジ）の配列が格納されている。含まれるデータは以下の通りである。

- **Study ノード**： 各研究を表すエンティティ
- **生物種メタデータ**： 実験対象生物の分類情報
- **MeSH アノテーション**： 医学主題標目（Medical Subject Headings）に基づく疾患・概念への紐付け
- **GO（Gene Ontology）アノテーション**： 遺伝子機能・生物学的プロセスへの紐付け
- **解析グループ情報**： 実験データの分析グループ構成およびファイル名
- **解析種別情報**： LC-MS、GC-MS、CE-MS 等の分析手法の種別

### 3-3. 変換処理の詳細

スクリプトは入力 JSON グラフを走査し、以下の処理を順次実施する。

**(a) Study ノードの RDF リソース化**  
各 Study ノードを識別し、Study 識別子あるいは Nodeid を主キーとして RDF URI を生成する。

**(b) LLM 由来テキストフィールドの付与**  
研究概要（summary）、観察事項（observations）、所見（findings）といった、大規模言語モデル（LLM）によって生成されたテキストフィールドを RDF リテラルとして付与する。

**(c) カテゴリ情報のトリプル化**  
各研究の分野区分として、branch（大分野）、category（分野）、subcategory（細分野）のヒエラルキー的カテゴリ情報を RDF プロパティとして付与する。

**(d) 外部データベースへのリンク付与**  
以下の外部生命科学データベースへの URI リンクを生成し、Linked Open Data 原則に則った知識グラフを構築する。

- MeSH（医学主題標目）
- GO（遺伝子オントロジー）
- NCBI Gene（遺伝子データベース）
- OMIM（オンラインメンデル遺伝データベース）

**(e) 生物種アノテーション**  
実験対象生物の種情報を RDF クラス・プロパティとして付与する。

**(f) 分析手法の RDF 語彙へのマッピング**  
LC-MS（液体クロマトグラフィー質量分析）、GC-MS（ガスクロマトグラフィー質量分析）、CE-MS（毛細管電気泳動質量分析）等、メタボロミクス研究固有の計測手法を所定の RDF 語彙にマッピングする。

### 3-4. 出力ファイル

スクリプトの実行により、以下のファイルが生成される。

| ファイル | 形式 |
|---|---|
| `<出力ベース名>.ttl` | Turtle 形式の RDF インスタンスデータ |
| `<出力ベース名>.nt` | N-Triples 形式の RDF インスタンスデータ |
| `<出力ベース名>.jsonld` | JSON-LD 形式の RDF インスタンスデータ |
| `<出力ベース名>_ontology.ttl` | Turtle 形式のオントロジー定義ファイル |

## 4. 主要スクリプト2：`ttl_to_rdfconfig.py`

### 4-1. 処理の概要

本スクリプトは、上記の変換処理で得られた Turtle（TTL）ファイル群（インスタンスデータおよびオントロジー）を入力として、RDF-config 形式の YAML 設定ファイル群の初期草案を自動生成するツールである。RDF-config は、SPARQL エンドポイントとその利活用インターフェースを記述するための設定フォーマットであり、本スクリプトはその人手による記述コストを大幅に低減することを目的としている。

### 4-2. 処理ロジックの詳細

本スクリプトの処理は、以下の5段階から構成される。

**(1) Prefix の抽出と整序**  
オントロジー TTL とデータ TTL の双方を読み込み、`@prefix` 宣言の順序を可能な限り保持しながら名前空間エイリアスを収集する。同一の名前空間 URI に複数のエイリアスが存在する場合、データ TTL 側で使用されているエイリアスを優先して採用する。

**(2) 主語クラスの発見**  
オントロジー内の `owl:Class` として定義されたリソースを検出し、データ TTL において `rdf:type` トリプルを通じて実際に使用されているクラスのみを対象として採用する。クラスのローカル名から CamelCase 形式のサブジェクト名を自動生成する。

**(3) 述語・目的語の生成**  
各クラスのインスタンスに対して実際に使用されている述語を収集し、目的語が別のクラスの型付きリソースを参照する場合には対象クラスを推論する。述語・目的語名は重複を回避しながら snake_case 形式で自動生成される。

**(4) カーディナリティの推論**  
データ TTL に基づいて各述語の出現頻度を実測し、以下の記号で記述量を表現する。なお、これらの値は供給データに基づく実証的な推定値であり、形式的なオントロジー保証ではない。

| 記号 | 意味 |
|---|---|
| `?` | 0 または 1 回 |
| `*` | 0 回以上 |
| `+` | 1 回以上かつ複数観測あり |
| `{n}` | 常に n 回 |
| `{n,m}` | n 回以上 m 回以下 |

**(5) `description.yaml` の生成**  
オントロジーに記述された `rdfs:comment`、`rdfs:label`、あるいはローカル名から生成したフォールバック文言を優先順位に従って取得し、`description.yaml`（説明用 YAML）の初期草案を自動生成する。
